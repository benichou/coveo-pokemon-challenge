# ML Models — Pokemon Challenge

This document captures **what ML models we've added to the Coveo org, why each one is there, how we created them, and how they're wired into the search pipeline**. It's the single source of truth for the org's ML stack; the org configuration itself is the operational reality, but this file explains the *intent* behind it.

## In plain English

Coveo offers a family of optional Machine Learning models that sit on top of the index. By themselves they're inert — they have to be (a) created, (b) trained ("built"), and (c) *associated* with a query pipeline before they affect search. For this project we deployed two of them:

- **Relevance Generative Answering (RGA)** — the LLM-powered feature that writes a one-paragraph natural-language answer above the result list, grounded in the indexed content with citations. This is the "AI answer" experience.
- **Semantic Encoder (SE)** — a vector-embedding model that retrieves content by semantic meaning, not just keyword match. It feeds *better chunks* into RGA at query time, which makes RGA's answers better. It also improves regular search (non-RGA queries) on its own.

We chose these two because **Doc 2's Advanced tier explicitly asks for RGA**, and Coveo's own Admin Console actively recommends pairing RGA with SE for higher answer quality. Together they're the highest-leverage ML investment we can make on this org.

## The models we deployed

### `pokemon-rga` — Relevance Generative Answering

**What it does.** At query time, RGA retrieves the top-N most relevant items from the index, feeds their body content (in chunks) to an LLM as context, and returns a generated paragraph that synthesizes those chunks. Every claim in the answer is linked back to the source document(s) as a citation. The user can give 👍/👎 feedback, which loops back into the model.

**Why we chose it.**
- Doc 2 Advanced tier: *"Deploy Coveo RGA to get a generative experience."* It's explicitly required for Full ambition.
- It's the highest-impact panel demo — turning a results list into a written answer is the most visible product change.
- For Pokemon Q&A queries (*"what type is Charizard?"*, *"which Gen 2 Pokemon are part-Dark?"*), generative answers add real value over a result page the user has to read manually.
- It positions us as understanding Coveo's 2026 "AI-grounded search" story, not just the legacy search-UX story.

**What it learns from.** Item body content only — the text mapped to Coveo's `body` field. Not metadata fields (`pokemon_type`, `dex_number`, etc.). This means **scraping body quality directly determines RGA answer quality**. Our scraping config strips header/footer/ads and keeps the article text, which is what RGA needs.

**Build cost.** Coveo's wizard says *"a few minutes to several hours."* For our ~1,028 items, expect 30 min – 2 hours.

### `pokemon-se` — Semantic Encoder

**What it does.** SE encodes both queries and documents into vector embeddings. At query time, it re-ranks retrieval based on *semantic similarity* (matching by meaning) rather than just keyword overlap. So a query like *"fast electric mouse"* can surface Pikachu without those exact words appearing in the page; *"psychic legendary"* can surface Mewtwo, Mew, Lugia, etc.

**Why we chose it.**
- Coveo's own UI banner recommends pairing SE with RGA: *"RGA and SE models make a great team. Associate the 2 models to the same query pipeline to improve the relevance of answers generated by RGA."*
- The retrieval that RGA generates from is only as good as the chunks it gets fed. SE improves that retrieval, which compounds RGA's value.
- SE also helps non-RGA queries (regular result-list searches) by making the ranking more semantic.

**How it pairs with RGA.** They don't talk to each other directly. They're both associated with the same query pipeline. At query time:
1. Query enters the pipeline.
2. SE provides semantic-aware ranking — the result set ordered by *meaning relevance*, not just keyword overlap.
3. RGA reads the top-ranked items and generates the natural-language answer from them.

So SE is upstream of RGA in the data flow, but both live in the same pipeline, independently associated.

---

## Why these two together (panel narrative)

If the panel asks *"why both models, and in what order?"*, the answer is:

```
Query ─► Default Query Pipeline
            ├─ Semantic Encoder (re-ranks by meaning)
            └─ Relevance Generative Answering (generates answer from top items)
                                                     │
                                                     ▼
                                         Answer + citations + ranked results
```

SE and RGA are independent **builds** but compounding **runtime contributors**. We could have shipped RGA alone, but the answers would be weaker because RGA would be working with keyword-only retrieval. Adding SE makes the retrieval semantic-aware, which makes the answer better. Coveo's UI surfaces this pairing as a recommendation, and our build reflects that recommendation.

It also positions us as understanding the full ML stack instead of cherry-picking the most visible feature.

## How we created them (the Console steps)

Both models were created through the Coveo Admin Console, NOT through the REST API. Coveo's public docs (`docs.coveo.com/en/nb6a0085/`) only describe Console creation for RGA — the generic ML models API (`POST /machinelearning/model`) exists but RGA's required `engineId` value isn't published. We confirmed this by searching the official docs (see also the back-and-forth in `plans/coveo/pokemon-challenge-plan.md`).

The Console wizard steps were identical for both models:

1. **Coveo Admin Console** → **AI & ML** → **Models** → **Add model**.
2. Click the model card:
   - **`Relevance Generative Answering`** for `pokemon-rga`
   - **`Semantic Encoder`** for `pokemon-se`
3. **Step 1 — Learn from** (data scope):
   - **Sources:** `pokemondb-sitemap` (we'll add `pokemondb-push` once Phase 4 lands; RGA/SE can have sources added after creation).
   - **Filter mode:** Standard (no Coveo query syntax filters).
   - **Filters:** none.
   - **Click Apply.** Verify the right-side panel populates with Total items ≈ 1,028 and Candidates for learning is close to that number.
   - **Click Next.**
4. **Step 2 — Name and project:**
   - **Name:** `pokemon-rga` / `pokemon-se`
   - **Project:** skipped (we don't use Coveo projects).
   - **Click Start building.**
5. Confirm model lands on the Models page with **Status: Build in progress**.

Both builds run independently in parallel; there's no dependency between them.

## How we wired them into the query pipeline (the script)

Coveo separates model definitions from model activation. A built model does nothing until it's *associated* with a query pipeline. The association IS publicly API-documented, so we automated it: `scripts/ml/associate_models.sh`.

```
scripts/ml/associate_models.sh             # idempotent, applies both associations
scripts/ml/associate_models.sh --dry-run   # shows what would change
```

What the script does:

1. **Lists query pipelines** (`GET /rest/search/v1/admin/pipelines?organizationId={orgId}`) and finds the one with `isDefault: true`.
2. **Lists ML models** (`GET /rest/organizations/{orgId}/machinelearning/models`) and finds `pokemon-rga` and `pokemon-se` by their `modelDisplayName`. Each model record carries an `id` field (Coveo's docs sometimes call it `modelId`, but the actual response key is `id`) and an `engineId` — `genqa` for RGA and `embeddings` for SE.
3. **Reads existing associations** on the default pipeline (`GET /rest/search/v2/admin/pipelines/{id}/ml/model/associations` — note **v2**, not v1) — for idempotency, the script skips models already wired in. The response shape is `{"rules": [...], "totalEntries": N, "totalPages": N}`.
4. **Creates the missing associations** (`POST` same endpoint, body `{"modelId": "<id>", "useAdvancedConfiguration": false}`).
5. **Prints the final pipeline state** so you can verify the wiring is correct.

> **API surface notes** (discovered during 2026-05-30 implementation):
> - The pipelines list endpoint is **v1** (`/rest/search/v1/admin/pipelines`).
> - The ML associations endpoint is **v2** (`/rest/search/v2/admin/pipelines/{id}/ml/model/associations`).
> - Different Coveo APIs use different versions for different resources — no global v1/v2 split. Each endpoint has to be probed independently.
> - The ML models list returns each model's identifier as `id`, not `modelId`. Stale docs say `modelId` for the response; the field name in the actual JSON is `id`.

Re-running the script is safe — it detects existing associations and skips them, so it's a no-op once everything is in place.

> **Why we script this even though Console works.** Two reasons. First, reproducibility: if someone clones this repo to a fresh org, running this script (plus the Console-only model creation) is faster and less error-prone than clicking through the UI. Second, panel artifact: it demonstrates that we know which parts of Coveo's API surface are public and exercise them deliberately, instead of treating the platform as Console-only.

---

## Required API key privileges

`scripts/ml/associate_models.sh` uses `COVEO_ADMIN_API_KEY`. That key needs:

- **Search → Query pipelines → Edit** (to list pipelines and create associations)
- **Machine Learning → Models → View** (to look up model IDs by name)

These are documented in `docs/api-keys.md` as part of the `pokemon-source-admin` key's full privilege set. If the script fails with HTTP 403, your admin key is missing one of these and needs to be re-minted (Coveo enforces immutable privileges post-creation, so updating isn't an option).

## Active state checks

After running the association script, you can confirm the wiring two ways:

**Console:** Admin Console → Search → Query Pipelines → click `default` → Machine Learning tab. Both `pokemon-rga` and `pokemon-se` should appear.

**API (curl):**

```bash
curl -sS -H "Authorization: Bearer $COVEO_ADMIN_API_KEY" \
  "https://platform.cloud.coveo.com/rest/search/v2/admin/pipelines/<PIPELINE_ID>/ml/model/associations?organizationId=$COVEO_ORG_ID" \
  | python3 -m json.tool
```

You should see two entries under `rules`, one per model.

A model that's `BUILDING` can be associated, but it won't actually run on queries until it transitions to `READY`. So associations made now will start working automatically when the builds finish.

## Maintenance

- **Rebuild a model** (e.g., after adding `pokemondb-push` as a second source): Admin Console → Models → click the model name → Rebuild button. Or via API: `POST /rest/organizations/{orgId}/machinelearning/models/{modelId}/build`.
- **Add a new source to an existing model:** Admin Console → click the model → Edit → Add source → Save. The model needs to be rebuilt afterward.
- **Change RGA's prompt instruction:** Admin Console → click the model → edit "Prompt enhancement" field → Save. No rebuild needed for this change.
- **Disassociate a model from a pipeline:** Admin Console → Query Pipelines → default → Machine Learning tab → trash icon. Or via API: `DELETE /rest/search/v1/admin/pipelines/{pipelineId}/ml/model/associations/{associationId}`.

## What's deliberately NOT here

- The model IDs themselves (`benichouu9fose4g_rga_...`). They live in the Coveo org and are looked up by name in the script. If we needed them in `.env`, we'd add a `COVEO_RGA_MODEL_ID` variable — but the lookup is fast and means the script works on any org without env edits.
- The query pipeline ID. Same reason: looked up dynamically by `isDefault=true`.
- Why we chose Coveo's hosted LLM vs. bring-your-own. Not a current decision point; Coveo runs the LLM behind RGA and we don't get to swap it. If that ever changes, this section will too.

## Glossary

| Term | What it means |
|---|---|
| **Association** | The wiring that connects an ML model to a query pipeline. Without it, the model exists but does nothing at search time. |
| **Build** | The Coveo-side training process that turns a model definition into a usable model. Takes minutes to hours. |
| **Embedding** | A vector representation of a piece of text — a list of numbers that captures its meaning. Both RGA and SE create embeddings for indexed content during their builds. |
| **Engine** | The Coveo-internal identifier for a model type. The Console hides this; the REST API exposes it as `engineId`. For RGA, the public docs don't publish the engineId. |
| **Pipeline** (Query pipeline) | The path a search query takes inside Coveo. Rules, conditions, and ML models are all applied via the pipeline. Most orgs have one default pipeline plus a few targeted ones. |
| **Prompt enhancement** | A free-text instruction (up to 2,000 chars) you add to RGA telling the LLM how to behave (tone, audience, restrictions). Optional. |
| **RAG** | Retrieval-Augmented Generation — the pattern of "search first, then have an LLM write an answer from those results." RGA is Coveo's branded RAG. |
| **Semantic Encoder** | A Coveo ML model that creates embeddings for retrieval-time semantic ranking. Pairs naturally with RGA. |
